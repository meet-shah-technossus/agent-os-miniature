# Agent OS — POC Architecture Refactor: Implementation Plan v3

## Vision (Revised)

A **proof-of-concept** autonomous SDLC engine that takes requirements as input and produces a working single-module project through iterative prompt generation → code generation → code review cycles, with human-in-the-loop at two critical checkpoints. The scope is narrowed from a multi-module enterprise system to a single-project, single-pass POC.

---

## New Architecture Overview

### Components

| # | Component | Type | Role |
|---|-----------|------|------|
| 1 | **Orchestrator** | Simple Python script | Central coordinator — routes data between components, no git/code processing |
| 2 | **Prompt Generator** | OpenAI API call | Generates implementation/fix prompts from requirements or review JSON |
| 3 | **Code Generator** | CLI terminal agent | Generates code, creates repo, pushes to branches, raises PRs, resolves PR comments |
| 4 | **Code Reviewer** | CLI terminal agent | Reviews PRs on GitHub (no local code access), adds line/global comments, generates review JSON, pushes to main on final iteration |

### Removed Components (vs. v2)

- ~~Module Maker~~ — no multi-module decomposition needed for single-project POC
- ~~Validation Layer~~ — replaced by CI pipeline Python script generated by code generator
- ~~Agent Communication Bus~~ — orchestrator handles all coordination directly
- ~~Complex state machine~~ — simplified to linear iteration loop

## New Workflow

```
┌─────────────────────────────────────────────────────────────────────┐
│                         ORCHESTRATOR                                │
│                                                                     │
│  ┌──────────┐    ┌──────────────────┐    ┌───────────────────┐      │
│  │ Ingest   │───▶│ Prompt Generator │───▶│  Code Generator   │      │
│  │ Req/JSON │    │   (OpenAI API)   │    │  (CLI Terminal)   │      │
│  └──────────┘    └──────────────────┘    └───────────────────┘      │
│       ▲                                          │                  │
│       │           ┌──────────────────┐           │                  │
│       └───────────│  Code Reviewer   │◀──────────┘                  │
│                   │  (CLI Terminal)  │                               │
│                   └──────────────────┘                               │
└─────────────────────────────────────────────────────────────────────┘
```

### Detailed Flow

#### Iteration 1 (Initial Generation)
1. **Orchestrator** ingests requirements (`.txt`, `.csv`, `.xlsx`, JIRA, Asana, ADO)
2. **Orchestrator** → **Prompt Generator**: passes requirements
3. **Prompt Generator** generates implementation prompt via OpenAI API → writes to fixed local file
4. **🔁 HITL Checkpoint 1**: User reviews/edits the generated prompt, selects CLI tool, approves
5. **Orchestrator** → **Code Generator**: passes the (possibly edited) prompt
6. **Code Generator**:
   - Generates the entire project codebase
   - Creates a CI pipeline Python script (must NOT be in `.gitignore`)
   - Pushes code to **main** branch
   - Pushes code to **feature branch** (configurable name)
   - Raises a Pull Request from feature branch → main
7. **Orchestrator** → **Code Reviewer**: triggers review of the PR

#### Iteration 2+ (Fix Iterations)
1. **Code Reviewer**:
   - Reviews the PR code on GitHub (no local code access)
   - Adds line-specific comments on files in the PR
   - Adds global PR comments (folder structure, clean architecture, etc.)
   - Generates a structured review JSON with all comments/issues
2. **🔁 HITL Checkpoint 2**: User reviews the review JSON, approves
3. **Orchestrator** → **Prompt Generator**: passes the review JSON
4. **Prompt Generator** generates a fixes-only prompt from scratch using the review JSON → writes to fixed local file
5. **🔁 HITL Checkpoint 1**: User reviews/edits the fix prompt, selects CLI tool, approves
6. **Orchestrator** → **Code Generator**: passes the fix prompt
7. **Code Generator**:
   - Makes required code corrections
   - Resolves all PR comments from code reviewer
   - Pushes updated code to the **feature branch only** (NOT main)
   - CI pipeline script runs before each push

#### Final Iteration (Convergence)
1. **Code Reviewer** finds no issues → no comments needed
2. **Code Reviewer**:
   - Pushes code to **main** branch
   - Deletes the feature branch
   - Closes/merges the PR

### Branch Strategy
- Only **2 branches** visible at any time on GitHub: `main` and `{configurable-feature-branch}`
- Code generator pushes to main only in iteration 1
- Code generator pushes to feature branch in all iterations
- Code reviewer pushes to main only in the final iteration (when accepting)

### HITL (Human-in-the-Loop) Checkpoints
1. **After Prompt Generation** — user can view, edit, and approve the generated prompt; user selects which CLI tool to use
2. **After Code Review JSON** — user can view and approve the review JSON before it's fed back to prompt generator

### Phase 2: Backend — New Orchestrator & State Machine
**Goal**: Rewrite the orchestrator as a simple Python script with a linear iteration loop, new pipeline states, and new API routes.

#### 2.1 New Pipeline States
Replace the old complex state machine with a simplified set:
```python
class PipelineStatus(str, Enum):
    IDLE = "IDLE"
    LOADING_REQUIREMENTS = "LOADING_REQUIREMENTS"
    PROMPT_GENERATION = "PROMPT_GENERATION"
    HITL_PROMPT_REVIEW = "HITL_PROMPT_REVIEW"          # HITL checkpoint 1
    CODE_GENERATION = "CODE_GENERATION"
    CODE_REVIEW = "CODE_REVIEW"
    HITL_REVIEW_DECISION = "HITL_REVIEW_DECISION"      # HITL checkpoint 2
    PIPELINE_COMPLETE = "PIPELINE_COMPLETE"
    FAILED = "FAILED"
```

State transitions:
```
IDLE → LOADING_REQUIREMENTS → PROMPT_GENERATION → HITL_PROMPT_REVIEW
→ CODE_GENERATION → CODE_REVIEW → HITL_REVIEW_DECISION
→ PROMPT_GENERATION (loop) ... → PIPELINE_COMPLETE
```

#### 2.2 Orchestrator Rewrite (`agent_os/orchestrator/engine.py`)
- Strip out all module iteration logic, dependency DAG, multi-module sequencing
- Implement simple linear loop:
  1. Load requirements (or receive review JSON from iteration N-1)
  2. Call prompt generator
  3. Pause at HITL checkpoint 1
  4. On approval: call code generator with selected CLI tool
  5. Call code reviewer
  6. Pause at HITL checkpoint 2
  7. If review JSON has issues → loop back to step 2
  8. If no issues → mark PIPELINE_COMPLETE
- Remove `HandlerContext` dependencies on bus, rollback manager
- Remove module-based state tracking; track only: `current_iteration`, `pipeline_status`, `review_json_path`

#### 2.3 Simplified State Manager (`agent_os/orchestrator/state.py`)
- Rewrite `TRANSITIONS` dict to match new states
- Remove module-related fields from `PipelineState`
- Add `current_iteration: int` tracked at orchestrator level

#### 2.4 New Orchestrator API Routes (`agent_os/api/routes/orchestrator.py`)
Replace the old `pipeline.py` with new routes:
- `GET /api/orchestrator/status` — current iteration, pipeline status, active component
- `POST /api/orchestrator/start` — begin pipeline from requirements
- `POST /api/orchestrator/approve-prompt` — HITL checkpoint 1: approve the generated prompt (with optional edits + selected CLI tool)
- `POST /api/orchestrator/approve-review` — HITL checkpoint 2: approve the review JSON
- `POST /api/orchestrator/pause` — pause the pipeline
- `POST /api/orchestrator/reset` — reset to IDLE
- `GET /api/orchestrator/iterations` — list all iterations with their status, prompt, review JSON

#### 2.5 Configuration Updates
- Add to `config.yaml` / `AgentOSConfig`:
  - `project.repo_name: str` — GitHub repository name (local folder matches this)
  - `project.feature_branch: str` — configurable feature branch name (default: `dev`)
  - `project.prompt_file_path: str` — fixed location for the generated prompt file
- Remove `codex.model_routing` and `codex.cli_routing` `MODULE_MAKER` entries
- Simplify `orchestrator` config: remove `max_iterations_per_module` (rename to `max_iterations`), remove module-specific settings

#### 2.6 Storage Schema Updates
- Rewrite `iterations` table: remove `module_id` dependency, add `review_json_content`, `prompt_content`, `cli_tool_used`, `ci_result`
- Remove `modules` table (or keep empty for backward compat)
- Update `pipeline_state` table: remove `current_module_id`, keep `current_iteration`

**Expected outcome**: Backend compiles with new simplified orchestrator, new API routes, new state machine. Pipeline can be started but prompt/code/review generation not yet functional.

  - Iteration 2+: Generate fixes-only prompt from review JSON (from scratch, not incremental)

  - Return the generated prompt text

#### 3.2 Code Generator Enhancements (`agent_os/code_generator/`)

  - **Iteration 1 — new responsibilities**:
    1. Generate entire project codebase from prompt
    2. Generate CI pipeline Python script (`ci_pipeline.py` or similar) — the script must test that the build succeeds before every push; ensure this file is NOT in `.gitignore`
    3. Initialize git repo if needed
    4. Push code to `main` branch
    5. Push code to feature branch (configurable name)
    6. Raise a Pull Request from feature branch → main
  - **Iteration 2+ — new responsibilities**:
    1. Make code corrections per the fix prompt

    3. Run CI pipeline script before pushing
    4. Push updated code to feature branch only (NOT main)
- **Update** `guardrails.py`:
  - Update guardrails to reflect new responsibilities (CI script generation, branch management, PR handling)
  - Add explicit guardrail: "Do NOT push to main after iteration 1"
  - Add guardrail: "Generate CI pipeline Python script in iteration 1"
  - Add guardrail: "Resolve all open PR comments before pushing"
- **Update/Add** git integration:
  - Use `agent_os/git_ops/manager.py` for branch operations
  - Use `agent_os/github/client.py` for PR creation and comment resolution
- **Update** `completion.py`: Add completion detection for git push + PR creation
- **VCS Provider conditional** (see Section 3.5):
  - Read `config.requirements_source` at runtime to determine VCS provider
  - If `requirements_source == "ado"` → use `ADOClient` for all git/VCS operations:
    - Repo creation: ADO Git Repositories API (`POST /{org}/{project}/_apis/git/repositories`)
    - Git remote URL: `dev.azure.com/{ado_org}/{ado_project}/_git/{repo_name}`
    - PR creation: ADO Pull Requests API (`POST .../pullrequests`)
    - PR comment resolution: ADO Threads API (`PATCH .../threads/{id}` with `status: "fixed"`)
  - If any other source (JIRA, Asana, device files, etc.) → use `GitHubClient`:
    - Repo creation: GitHub Repositories API (`POST /user/repos`)
    - Git remote URL: `github.com/{owner}/{repo_name}.git`
    - PR creation: GitHub Pull Requests API (`POST /repos/{owner}/{repo}/pulls`)
    - PR comment resolution: GitHub PR Comments API (`PATCH .../pulls/comments/{id}` as resolved)
  - All operations routed through the `VCSClient` abstraction — `runner.py` does not branch on provider type directly

#### 3.3 Code Reviewer Enhancements (`agent_os/code_reviewer/`)
- **Rewrite** `runner.py`:
  - **Input**: PR URL/number (not local codebase — reviewer has NO local code access)
  - **Operations**:
    1. Fetch PR diff from VCS provider API (GitHub or ADO — see Section 3.5)
    2. **Run FULL existing 15-point checklist review** (ALL criteria preserved from current `runner.py`, none removed):
       - **Code Correctness** — logic errors, bugs, edge cases, unexpected behaviour
       - **Readability & Clarity** — naming conventions, code formatting, inline comments
       - **Code Structure & Design** — modularity, SOLID principles, separation of concerns
       - **Performance & Efficiency** — algorithmic complexity, unnecessary loops, memory usage
       - **Security** — injection vulnerabilities, auth weaknesses, data exposure, OWASP Top 10
       - **Error Handling** — exception coverage, fallback logic, graceful degradation
       - **Code Standards** — style guide conformance, linting, formatting consistency
       - **Testing & Coverage** — unit/integration test presence, test quality, coverage percentage
       - **Documentation** — docstrings, README accuracy, inline comments for complex logic
       - **Maintainability** — tech debt, code duplication, excessive coupling, fragility
       - **Dependencies & Imports** — unused imports, outdated dependencies, version pinning
       - **Logging & Monitoring** — structured logging, error tracing, observability
       - **Version Control** — commit message quality, branch hygiene, merge readiness
       - **UI/UX** — usability, accessibility, responsive design (only when frontend files changed)
       - **Overall Impact** — risk assessment, regression potential, release readiness
    3. **ADDITIONALLY run these new checks ON TOP of the 15-point checklist:**
       - **File size rule**: Flag every file exceeding 200 lines (count lines per file in the PR diff)
       - **Clean architecture compliance**: Verify layer separation is respected (no business logic in routes/controllers, no direct DB calls from UI layer, no cross-layer bypasses)
       - **Folder structure review**: Verify every new/modified file is placed in the correct project directory per the project's declared conventions
    4. Apply **existing strict rejection rules** (unchanged from current implementation):
       - Syntax error present → **Rejected** immediately, no score assigned
       - Tests failing → convergence score capped at ≤ 40
       - Critical security vulnerability present → **Rejected** immediately
    5. Post **line-specific inline PR comments** for file-level findings (code correctness, readability, security, file-size violations) — via VCS provider API
    6. Post **global PR comments** for structural and architectural findings (folder structure issues, clean architecture violations, cross-cutting concerns) — via VCS provider API
    7. Generate structured review JSON aggregating ALL findings (line-specific + global + checklist scores)
    8. Write review JSON to the configured local file path for the prompt generator to consume
  - **Final iteration** (when `overall_status == "accepted"` and no open issues):
    1. Approve the PR
    2. Merge PR to main branch
    3. Delete the feature branch
    4. All three operations use the VCS provider abstraction (GitHub or ADO — see Section 3.5)
- **Update** `schema.py`:
  - New `ReviewJSON` schema containing:
    - `overall_status`: `"needs_work"` | `"accepted"` | `"rejected"`
    - `checklist_scores`: dict mapping each of the 15 criteria names to a score 0–100
    - `overall_score`: int 0–100 (weighted average of checklist scores)
    - `line_comments`: list of `{file, line, comment, severity, checklist_item}`
    - `global_comments`: list of `{comment, category, severity}`
    - `folder_structure_issues`: list of `{path, issue, expected_location}`
    - `architecture_issues`: list of `{description, layer, severity}`
    - `file_size_violations`: list of `{file, line_count}` for files > 200 lines
    - `summary`: string — overall verdict and recommended next steps
  - The JSON must contain all info needed by prompt generator to generate a targeted iteration N+1 fix prompt
- **VCS Provider conditional** (see Section 3.5):
  - If `requirements_source == "ado"` → all PR operations (fetch diff, add comments, approve, merge, delete branch) use `ADOClient`
  - Otherwise → all PR operations use `GitHubClient`
- **Update** ceiling/skills: reviewer must NOT access local codebase; all code access via VCS provider PR diff only

#### 3.4 CI Pipeline Integration
- Code generator in iteration 1 must create a Python script that:
  - Validates the project builds successfully (e.g., `python -m py_compile`, `npm run build`, etc.)
  - Runs before every git push
  - Is NOT added to `.gitignore`
- This script path should be configurable or follow a convention (e.g., `ci_check.py` at project root)
- The CI pipeline script should be appropriate for the project language (Python, JS, etc.)

**Expected outcome**: All three backend components fully functional. Prompt generator uses OpenAI API. Code generator can create repos, push code, create PRs, resolve comments. Code reviewer can review PRs, add comments, generate JSON, and merge on final iteration.

#### 3.5 VCS Provider Abstraction (`agent_os/vcs/`)
**Goal**: Route all git/PR operations to either GitHub or Azure DevOps based on the configured requirements source, without the generator/reviewer runners knowing which provider is active.

**Runtime conditional** (checked by orchestrator when initialising each runner):
```python
if config.requirements_source == "ado":
    vcs_client = ADOClient(org=config.ado_org, project=config.ado_project, token=config.ado_token)
else:
    vcs_client = GitHubClient(owner=config.github_owner, token=config.github_token)
```

**New file** `agent_os/vcs/base.py` — abstract `VCSClient` with unified interface:
- `create_repo(repo_name)` → repo URL
- `get_remote_url(repo_name)` → git remote URL string passed to `git remote add origin …`
- `create_pr(repo, source_branch, target_branch, title, body)` → PR ID/number
- `add_line_comment(repo, pr_id, file_path, line, comment)` → comment/thread ID
- `add_global_comment(repo, pr_id, comment)` → comment/thread ID
- `resolve_comment(repo, pr_id, comment_id)` → None
- `merge_pr(repo, pr_id)` → None
- `delete_branch(repo, branch_name)` → None
- `get_pr_diff(repo, pr_id)` → raw diff string

**New file** `agent_os/vcs/github_client.py` — `GitHubClient(VCSClient)`:
- Auth: `Authorization: token {github_pat}` header; base URL `https://api.github.com`
- `create_repo`: `POST /user/repos` (or `/orgs/{org}/repos`)
- `get_remote_url`: returns `https://github.com/{owner}/{repo}.git`
- `create_pr`: `POST /repos/{owner}/{repo}/pulls`
- `add_line_comment`: `POST /repos/{owner}/{repo}/pulls/{pr}/comments` (body includes `path`, `line`)
- `add_global_comment`: `POST /repos/{owner}/{repo}/issues/{pr}/comments`
- `resolve_comment`: `PATCH /repos/{owner}/{repo}/pulls/comments/{id}` — mark thread resolved
- `merge_pr`: `PUT /repos/{owner}/{repo}/pulls/{pr}/merge`
- `delete_branch`: `DELETE /repos/{owner}/{repo}/git/refs/heads/{branch}`
- `get_pr_diff`: `GET /repos/{owner}/{repo}/pulls/{pr}` with `Accept: application/vnd.github.diff`

**New file** `agent_os/vcs/ado_client.py` — `ADOClient(VCSClient)`:
- Auth: `Authorization: Basic {base64(":" + ado_pat)}` header
- `create_repo`: `POST https://dev.azure.com/{org}/{project}/_apis/git/repositories`
- `get_remote_url`: returns `https://dev.azure.com/{org}/{project}/_git/{repo}`
- `create_pr`: `POST .../git/repositories/{repoId}/pullrequests`
- `add_line_comment`: `POST .../pullrequests/{prId}/threads` with `threadContext` (filePath + rightFileStart/End)
- `add_global_comment`: `POST .../pullrequests/{prId}/threads` without `threadContext`
- `resolve_comment`: `PATCH .../pullrequests/{prId}/threads/{threadId}` with `{"status": "fixed"}`
- `merge_pr`: `PATCH .../pullrequests/{prId}` with `{"status": "completed", "lastMergeSourceCommit": {...}}`
- `delete_branch`: `DELETE .../git/repositories/{repoId}/refs` body `[{"name": "refs/heads/{branch}", "newObjectId": "0000000000000000000000000000000000000000", "oldObjectId": "{currentTipSha}"}]`
- `get_pr_diff`: `GET .../pullrequests/{prId}/iterations/{iterationId}/changes`

**Key differences between providers:**

| Operation | GitHub | Azure DevOps |
|-----------|--------|--------------|
| Create repo | `POST api.github.com/user/repos` | `POST dev.azure.com/{org}/{project}/_apis/git/repositories` |
| Git remote URL | `github.com/{owner}/{repo}.git` | `dev.azure.com/{org}/{project}/_git/{repo}` |
| Create PR | `POST /repos/{owner}/{repo}/pulls` | `POST .../pullrequests` |
| Inline comment | `POST .../pulls/{pr}/comments` (`path` + `line`) | `POST .../pullrequests/{pr}/threads` (`threadContext` with `filePath` + line) |
| Global comment | `POST .../issues/{pr}/comments` | `POST .../pullrequests/{pr}/threads` (no `threadContext`) |
| Resolve comment | `PATCH .../pulls/comments/{id}` → resolved | `PATCH .../threads/{id}` → `status: "fixed"` |
| Merge PR | `PUT .../pulls/{pr}/merge` | `PATCH .../pullrequests/{pr}` (`status: "completed"`) |
| Delete branch | `DELETE .../git/refs/heads/{branch}` | `DELETE .../refs` (body with `newObjectId` all zeros + current tip SHA) |
| Auth format | `Authorization: token {pat}` | `Authorization: Basic {base64(":pat")}` |
| Comment model | Separate review comments vs issue comments | Unified **threads** (with optional file position) |
| Resolve model | `resolved: true/false` per comment | `status: active / fixed / wontFix / closed / byDesign / pending` per thread |

**Credentials reuse**: ADO credentials (`ado_org`, `ado_token`, `ado_project`) configured in the Requirements tab are **reused automatically** for VCS operations — no separate VCS credential fields are added. See Section 7.3 for the Settings UI note.

## Risk Notes

1. **OpenAI API key dependency** — Prompt generator now requires a valid OpenAI API key for every run (no fallback to CLI)
2. **VCS token dependency** — Code generator and reviewer both require VCS API access for all PR operations. If source is non-ADO: a valid GitHub PAT with `repo` scope is required. If source is ADO: a valid ADO PAT with **Code (Read & Write)** and **Pull Request Contribute** scopes is required.
3. **CLI tool availability** — The selected CLI tool must be installed and authenticated (already handled by Settings AI Tools tab)
4. **Single-project scope** — This POC does not support multi-module projects; attempting a complex project may exceed the capability of a single generation pass
5. **Branch conflicts** — If the user manually pushes to the feature branch outside the pipeline, merge conflicts may occur
6. **CI script quality** — The auto-generated CI pipeline script quality depends on the code generator's ability to understand the target framework's build process
7. **ADO auth complexity** — Azure DevOps PAT must be Base64-encoded as `:{pat}` (colon-prefixed) for the Basic auth header. Incorrect encoding silently returns HTTP 401. Test the PAT manually before running the pipeline.
8. **ADO branch deletion requires tip SHA** — Deleting a branch in ADO via the Refs API requires the current branch tip commit SHA (`oldObjectId`) in the request body. The code generator must capture and store this SHA after each push; failure to do so blocks branch cleanup at the final iteration.
