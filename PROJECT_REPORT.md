## 1. Project Vision & Purpose

**Agent OS** is an autonomous Software Development Lifecycle (SDLC) engine. It takes structured requirements — formatted as epics → features → stories → acceptance criteria — and produces production-ready software codebases through iterative cycles of:

1. **Prompt Generation** — converting requirements or review feedback into precise, actionable coding prompts
2. **Code Generation** — executing those prompts using AI coding CLI tools (Codex, Claude Code, Gemini CLI, etc.)
3. **Code Review** — reviewing the generated code via Pull Request diffs and producing structured feedback JSON
4. **Iteration** — looping until the code meets all acceptance criteria and passes all review gates

The system is designed to require **minimal human intervention**, with Human-in-the-Loop (HITL) checkpoints at critical decision points so the user can review, edit, and approve before the pipeline continues.

**Core design goal**: A user should be able to paste requirements into the system, click "Start", and receive a working, reviewed, committed, PR-raised codebase — with optional oversight at every step.

## 2. Architecture Evolution

### v1 — Initial Plan (Multi-Module Enterprise System)

The first design was ambitious: a fully parallel multi-agent system for multi-module enterprise projects.

**Key components planned:**
- **Module Maker** — decomposed requirements into structured module JSON with dependency DAGs
- **Agent Communication Bus** — pub/sub message passing system with typed channels (`module_updates`, `prompt_ready`, `generation_status`, `validation_results`, `review_feedback`, `hitl_requests`, etc.)
- **Validation Layer** — separate subprocess running ruff, mypy, pytest, bandit
- **Module-by-module pipeline** with per-module state machines
- Complex orchestrator state machine with many states

**Why it was changed:**  
The v1 architecture was over-engineered for the proof-of-concept stage. A full multi-agent bus, module decomposer, and parallel validation layer added enormous complexity before the core loop (prompt → code → review) was even proven to work. The decision was made to scope down to a single-module POC.

---

### v2 — Code Quality Hardening Layer Added

Before the POC refactor, v2 added a parallel communication architecture where all agents communicated simultaneously through a shared message bus. Every message had:
- A `channel` identifier
- `sender` agent name
- `timestamp`
- `module_id` and `iteration` context
- `correlation_id` for lifecycle tracing

**10 defined channels**: `module_updates`, `prompt_ready`, `generation_status`, `validation_results`, `review_feedback`, `hitl_requests`, `hitl_responses`, `pipeline_events`, `error_alerts`, `agent_heartbeats`

This architecture was documented but ultimately simplified in v3 — the orchestrator took on direct coordination without a formal bus.

---

### v3 — POC Architecture (Current Implementation)

The third iteration refocused on **proving the core loop works** before adding enterprise-scale complexity.

**What was removed from earlier versions:**
- Module Maker component entirely
- Validation Layer (replaced by CI pipeline script generated by the code generator)
- Agent Communication Bus (orchestrator handles coordination directly)
- Complex multi-module state machine

**What was kept and simplified:**
- Orchestrator as a simple linear coordinator (no intelligence — just routes data)
- Prompt Generator (OpenAI API direct call)
- Code Generator (CLI terminal agent)
- Code Reviewer (CLI terminal agent, PR-based)
- Human-in-the-Loop at 2 critical checkpoints

**New additions in v3:**
- Two pipeline modes: **Standard** and **GitHub Review**
- Story queue system for GitHub Review mode
- Requirements ingestion from multiple sources (Device, JIRA, Asana, ADO)
- Full React frontend with 9 views
- CLI tool management with 7 supported tools

    "github_token": "GITHUB_TOKEN",
    "ollama_base_url": "OLLAMA_BASE_URL",

`mask_secret()` — masks secrets for safe display (shows first 3 + `...` + last 2 chars)  
`resolve_secret()` — resolves a secret through the full priority chain  

**`loader.py`** — Loads `config.yaml` from disk; applies defaults via `get_default_config()`.

### 6.2 Orchestrator (`agent_os/orchestrator/`)

**`engine.py`** — `Orchestrator` class. The central state machine controller.

Key responsibilities:
- Holds `config: AgentOSConfig`
- Owns the `Database` and `StateManager`
- Runs the pipeline in a background thread (`_pipeline_thread`)
- Publishes events to the WebSocket queue (`_ws_queue`)
- Manages `_stop_event`, `_pause_event`, `_code_gen_stop_requested` threading events
- Exposes `start()`, `stop()`, `pause()`, `resume()` methods
- Auto-updates iteration records on state transitions

**`handlers.py`** — One function per pipeline state. Each handler receives a `HandlerContext` (config + db + state manager) and transitions to the next state when done:

- `handle_idle()` → transitions to `LOADING_REQUIREMENTS`
- `handle_loading_requirements()` → parses YAML, stores requirements, derives project name from epic titles
- `handle_prompt_generation()` → calls `PromptGeneratorRunner`
- `handle_hitl_prompt_review()` → pauses, waits for human approval
- `handle_code_generation()` → calls `CodeGeneratorRunner`
- `handle_code_review()` → calls `CodeReviewerRunner`
- `handle_hitl_review_decision()` → pauses, waits for human decision
- `handle_github_fork_clone()` → forks repo, clones locally, loads requirements into DB, creates branch
- etc.

**`state.py`** — `StateManager` persists `PipelineState` to SQLite. Manages state transitions and fires transition listeners.

### 6.3 Prompt Generator (`agent_os/prompt_generator/runner.py`)

`PromptGeneratorRunner` generates two types of prompts:

**Iteration 1 — Implementation Prompt**  
Takes raw requirements text → calls LLM with `_SYSTEM_IMPLEMENTATION` system prompt → streams response → writes to `config.project.prompt_file_path` (falls back to `data/prompts/latest.md`).

The system prompt instructs the LLM to:
- Be explicit about every file to create
- Specify full technology stack, versions, frameworks, directory structure
- Include a CI pipeline script (`ci_check.py`) that must NOT be in `.gitignore`
- Not invent requirements beyond what was provided
- Reason through implementation challenges before writing

**Iteration 2+ — Fix Prompt**  
Takes review JSON → calls LLM with `_SYSTEM_FIX` system prompt → streams response.

The fix prompt system instructs the LLM to:
- NOT start from scratch (existing project)
- Include "DO NOT TOUCH" sections for passing files
- Reference exact file paths and line numbers from review JSON
- Explain WHY each fix is needed
- Instruct the agent to run `ci_check.py` after all fixes

**LLM Routing** (`_stream_llm()`):  
Reads `config.prompt_generator.provider`:
- `"ollama"` → calls `_stream_ollama()` using `openai.OpenAI(api_key="ollama", base_url=f"{ollama_base_url}/v1")`
- `"openai"` → calls `_stream_openai()` using the standard OpenAI client

This means **both Ollama and OpenAI use the same OpenAI Python SDK** — Ollama implements the OpenAI-compatible `/v1/chat/completions` endpoint.

### 6.5 Code Reviewer (`agent_os/code_reviewer/runner.py`)

`CodeReviewerRunner`:
- Accesses code exclusively via **PR diffs** (no local file access)
- Posts line-specific comments on PR files via GitHub/ADO API
- Posts global PR comments for architectural issues
- Produces structured `ReviewJSON` with per-file issues, severity levels, and suggested fixes
- Applies a 15-point review checklist
- Severity levels: `critical`, `high`, `medium`, `low`
- High/critical severity always blocks acceptance

**`agent_os/code_reviewer/schema.py`** — `ReviewJSON` model:
```
{
  "overall_verdict": "needs_work" | "accepted",
  "issues": [
    {
      "file": "path/to/file.py",
      "line_start": 42,
      "line_end": 50,
      "severity": "high",
      "category": "security" | "architecture" | "correctness" | ...,
      "description": "...",
      "suggested_fix": "..."
    }
  ]
}
```

### Agent Summaries

**Orchestrator** — Simple Coordinator. Routes data only. No intelligence. State-faithful, HITL-respectful.

**Prompt Generator** — Technical Communication Specialist. Clarity above all. Contextually rich. Iteration-aware (full prompt for iteration 1, targeted fix prompt for iteration 2+). Structure-obsessed.

**Code Generator** — Senior Software Engineer + VCS Operator. Spec-faithful (only implements what is specified). Complete (no partial implementations). VCS-disciplined (pushes to main only on iteration 1, to feature branch on 2+). CI-first (runs `ci_check.py` before every push). PR-accountable (resolves all reviewer comments before pushing). ## 10. Pipeline State Machine

### Standard Mode States

```
IDLE
  → LOADING_REQUIREMENTS      (parse requirements YAML, store in DB)
  → PROMPT_GENERATION         (call Prompt Generator)
  → HITL_PROMPT_REVIEW        (pause, wait for human approval)
  → CODE_GENERATION           (call Code Generator CLI tool)
  → [CODE_GEN_FAILED]         (terminal state if generation fails)
  → [CODE_GEN_STOPPED]        (terminal state if user stopped)
  → CODE_REVIEW               (call Code Reviewer)
  → HITL_REVIEW_DECISION      (pause, wait for human decision)
  → [loop back to PROMPT_GENERATION if needs_work and iteration < max]
  → PIPELINE_COMPLETE         (accepted or max iterations reached)
  → [FAILED]                  (unrecoverable error)
```

### GitHub Review Mode States

```
IDLE
  → LOADING_REQUIREMENTS
  → ANALYSING_DEPENDENCIES    (LLM analyses story order/dependencies)
  → QUEUE_READY               (story queue built, first story dequeued)
  → STORY_PROMPT_GENERATION   (per-story prompt generation)
  → STORY_CODE_GENERATION     (per-story code generation)
  → STORY_CODE_REVIEW         (per-story PR-based review)
  → STORY_COMPLETE            (story accepted, dequeue next)
  → [loop per story]
  → PIPELINE_COMPLETE
```

### Convergence Rules

| Rule | Description |
|---|---|
| `no_high_severity` | Accept when no `high` or `critical` issues remain (default) |
| `no_critical` | Accept when no `critical` issues remain |
| `all_accepted` | Accept only when all issues are resolved |

## 11. GitHub Review Mode

This mode is for reviewing and improving an **existing codebase** hosted on GitHub, rather than generating a new one from scratch.

### How It Works

1. User provides a **Source Repository URL** (e.g. `https://github.com/org/repo`)
2. User provides requirements (from any source: ADO, JIRA, device, etc.) that define what the code should do
3. Pipeline forks the repo under the authenticated GitHub account
4. Clones the fork locally to `config.project.root_path`
5. For each story in the requirements:
   - Creates a branch `{prefix}{story-id}-{short-desc}` from `main`
   - Prompt Generator generates a prompt for this story's requirements
   - Code Generator makes changes on the branch
   - Code Reviewer reviews the PR diff, posts comments, produces ReviewJSON
   - Loop until accepted
   - Merge branch, delete it, move to next story

### Git Behaviour per Iteration

| Action | Iteration 1 | Iteration 2+ |
|---|---|---|
| Fork | Done at pipeline start | N/A |
| Clone | Done at pipeline start | N/A |
| Branch | `story-{id}-{desc}` from main | Reuse existing branch |
| Code changes | Generate from prompt | Apply fixes from review JSON |
| Push | Push to story branch | Push to story branch |
| PR | Create PR (branch → main) | Already exists |
| Comments | N/A | Resolve all existing comments |
| Merge | When accepted | When accepted → merge + delete branch |

### Story Queue

`StoryQueueManager` uses an LLM call to:
- Analyse all stories and their interdependencies
- Produce an ordered queue (topological sort by dependency)
- Store in `story_queue` SQLite table

Each story queue item: `id`, `pipeline_id`, `story_id`, `title`, `description`, `acceptance_criteria`, `position`, `status` (queued/in_progress/completed/failed), `branch_name`, `pr_number`, `pr_url`, `iteration_count`, `depends_on`, `created_at`, `completed_at`.

### GitHub Copilot OAuth — GITHUB_TOKEN Conflict

**Problem**: `gh auth status` reported "authenticated" via `GITHUB_TOKEN` env var even when the user had not completed the OAuth flow. This gave a false positive for Copilot auth status.

```python
def _run_no_gh_token(cmd: list[str], **kwargs) -> subprocess.CompletedProcess:
    env = {k: v for k, v in os.environ.items() 
           if k not in ("GITHUB_TOKEN", "GH_TOKEN")}
    return subprocess.run(cmd, env=env, **kwargs)
```

The OAuth login terminal command also prepends:
```powershell
Remove-Item Env:GITHUB_TOKEN -ErrorAction SilentlyContinue; Remove-Item Env:GH_TOKEN -ErrorAction SilentlyContinue;
```
(on Mac: `unset GITHUB_TOKEN GH_TOKEN;`)

## 14. VCS (Version Control) System

### VCS Provider Decoupling

**Before**: `vcs/factory.py` read `config.requirements.source` to decide GitHub vs ADO. This meant if you set ADO as your requirements source, all your git operations went to ADO too — even if your code was on GitHub.

**After**: A dedicated `VCSConfig` model was added with an explicit `provider` field. The factory now reads `config.vcs.provider`. The requirements source is now completely independent from the VCS target.

```python
# VCSConfig
class VCSConfig(BaseModel):
    provider: str = "github"   # "github" | "ado"
```

**Backward compatibility**: If `config.vcs` is absent (old configs), the factory falls back to checking `config.requirements.source`.

### GitHub VCS Client

Uses GitHub REST API v3:
- `POST /repos/{owner}/{repo}/forks` — fork a repo
- `POST /repos/{owner}/{repo}/git/refs` — create branches
- `POST /repos/{owner}/{repo}/pulls` — create PRs
- `GET /repos/{owner}/{repo}/pulls/{number}/files` — get PR diff
- `POST /repos/{owner}/{repo}/pulls/{number}/comments` — post line comments
- `POST /repos/{owner}/{repo}/issues/{number}/comments` — post global comments
- `PUT /repos/{owner}/{repo}/pulls/{number}/merge` — merge PR

### ADO VCS Client

Uses Azure DevOps REST API v7.1:
- Projects, repositories, pull requests, work items
- Basic auth via base64-encoded `:PAT`

### Fix 4: Ollama / Remote GPU Support for Prompt Generation

**Problem**: Prompt Generator only supported OpenAI API. Users with a local or remote GPU running Ollama had no way to use it.

**Why**: Reduce OpenAI API costs, allow offline/VPN-based GPU usage.

**Implementation**:

**Config schema** (`config/schema.py`):
```python
OLLAMA_MODELS = ["llama3.1:8b", "llama3.2:3b", "llama3:latest", "qwen2.5:7b", ...]

class OllamaConfig(BaseModel):
    base_url: str = "http://localhost:11434"
    model: str = "llama3.1:8b"
    timeout_seconds: int = 300

class PromptGeneratorConfig(BaseModel):
    provider: str = "ollama"   # "ollama" | "openai"
    ollama_model: str = "llama3.1:8b"
    openai_model: str = "gpt-4.1-mini"
```

**Env** (`.env`):
```
OLLAMA_BASE_URL=http://localhost:11434
```
(Can be set to remote GPU URL over VPN, e.g. `http://192.168.10.5:11434`)

**Runner** (`prompt_generator/runner.py`):
```python
def _stream_llm(self, messages, on_stdout):
    if self._config.prompt_generator.provider == "ollama":
        return self._stream_ollama(messages, on_stdout)
    return self._stream_openai(messages, on_stdout)

def _stream_ollama(self, messages, on_stdout):
    client = openai.OpenAI(
        api_key="ollama",   # Ollama ignores this but SDK requires it
        base_url=f"{self._config.ollama.base_url}/v1"
    )
    # Streams using the same OpenAI SDK interface
```

**Why this works**: Ollama implements the OpenAI-compatible `/v1/chat/completions` endpoint. The OpenAI Python SDK works against both backends without any code changes — just a different `base_url`.

**Frontend**: Added Prompt Generator section to Pipeline tab in Settings with provider toggle (Ollama / OpenAI), Ollama fields (base URL, model dropdown with all available models, timeout), and OpenAI model input.

### Fix 5: GitHub Test Button Using Stale Saved Token

**Problem**: The "Test" button next to the GitHub PAT input in Settings always tested the *saved* token from config/env, not the token the user had just typed. If a user changed their PAT in the input field and clicked Test, it still showed "Authenticated as [old username]".

**Root cause**: `POST /api/settings/test-github` read from `resolve_secret()` (config/env) and accepted no request body. `api.testGitHub()` sent no body. The UI token value was never transmitted.

**Fix**:

Backend (`schemas.py`) — Added `TestGitHubRequest`:
```python
class TestGitHubRequest(BaseModel):
    token: str = ""
```

Backend (`settings.py`) — `test_github_connection` now:
1. Accepts `body: TestGitHubRequest = TestGitHubRequest()`

### Fix 8: Native Dropdown Text Invisible in Dark OS Theme

**Problem**: All `<select>` dropdown menus (in SettingsView, AgentsView, CommandCenter, CreateAgentWizard) showed invisible text when the Windows OS was set to dark mode. The text only became visible on hover or after selection.

**Root cause**: The `<select>` element uses `bg-white/[0.04]` (nearly transparent) via Tailwind. On Windows dark mode, Chromium renders the native dropdown popup using Windows controls. Without `color-scheme: dark`, the browser could render dark background with dark text in the option list.

**Fix** (`index.css`):
```css

  color: var(--text-primary);
  background-color: var(--bg-surface);
}
select option {
  background-color: var(--bg-surface);  /* #12121c */
  color: var(--text-primary);            /* #e2e8f0 */
}
select option:hover,
select option:focus,
select option:checked {
  background-color: #6366f1;
  color: #ffffff;
}
```

This is a **global CSS rule** — it applies to all `<select>` elements across the entire application without changing any component code. Works in both light and dark OS themes.

## 19. Known Behaviour Details & Design Decisions

### Token Masking Strategy

Saved secrets are masked in the UI (showing `ghp_...Xz`). When the user saves without changing a token field, the frontend sends `''` (empty string). The backend treats `''` or `***` as "unchanged" and preserves the real token in memory. `_write_config_yaml()` only writes real (non-masked) tokens to YAML.

### Config Persistence

Settings are persisted to `config.yaml` by `_write_config_yaml()`. The YAML writer:
1. Reads the existing file first to preserve fields not exposed in the UI (e.g. `storage.db_path`, `codex.model_routing`, `budget.*`)
2. Overwrites only user-editable fields
3. Never writes secrets to YAML (writes empty strings for `openai_api_key`, `github_token` — real values stay in `.env`)
4. Conditionally writes `ado_token` only if it's a real (non-masked) value

### Pipeline Cannot Be Changed While Active

If the pipeline is not in IDLE state, the requirements endpoints return:
```json
{"detail": "Pipeline is active (state: HITL_REVIEW_DECISION). Reset or complete the pipeline before changing requirements."}
```

This prevents mid-run changes to the requirements file that would corrupt the in-progress iteration.

### Windows-Specific Handling

- PowerShell 5.1: no `&&` support → uses `;` for command chaining
- No PTY (pseudo-terminal) on Windows → subprocess streaming uses pipe mode instead
- OAuth commands: uses `Remove-Item Env:VARIABLE` instead of `unset VARIABLE`
- ADO MCP JSON path: uses `%APPDATA%\Claude\claude_desktop_config.json` instead of `~/Library/...`

### Ollama as OpenAI API Proxy

Ollama's HTTP server implements `/v1/chat/completions` exactly like OpenAI. By constructing the OpenAI Python client with `base_url=f"{ollama_url}/v1"` and `api_key="ollama"` (any string works — Ollama ignores it), streaming works identically. This means no separate Ollama SDK is needed.

### VCS Client Returns None Gracefully

`make_vcs_client()` returns `None` when credentials are absent (no token, no owner, no repo). All callers check for `None` before using the client, so missing VCS config doesn't crash the pipeline.
